1. Platform Overview
NOOG is a playlist-based adult video player that embeds videos from affiliate partner sites. It never hosts content. Revenue comes from affiliate commissions when users click through to partner sites.
Core constraint: Videos play inside cross-origin iframes. The browser blocks all communication between NOOG's page and the iframe content. NOOG cannot detect play, pause, ended, or timeupdate events from embedded players. This has been tested and confirmed across all partner embeds (FapHouse, BoyfriendTV, Faptor, XHamster, XVideos, Pornhub). None broadcast postMessage events.
Initial target audience: Gay male viewers.

3. Video Embed System
3.1 URL Resolution
Some source sites use redirect URLs. The backend resolves them to extract the actual video ID.
Resolution endpoint: GET /api/resolve-url?url=<encoded_url>
Flow:

Input: redirect URL (e.g., gaymaletube.com/out/?l=<base64blob>)
Backend decodes the base64 blob, strips binary junk, extracts destination URL (e.g., fhgte.com/videos/4rZCDE)
Returns the cleaned destination URL

3.2 Embed URL Transform
The getEmbedUrl(url) function transforms a resolved URL into an iframe-compatible embed URL.
Transform rules:
Source DomainInput PatternOutput Patternfaphouse.com / fhgte.com/videos/slug-{ID} or ?hash={ID}faphouse.com/embed/{ID}?autoplay=1&mute=1boyfriendtv.com / bfhub.com/videos/{ID}boyfriendtv.com/embed/{ID}?autoplay=1&mute=1faptor.com/videos/{ID}faptor.com/embed/{ID}?autoplay=1&mute=1pornhub.com/view_video.php?viewkey={ID}pornhub.com/embed/{ID}?autoplay=1&mute=1xhamster.com/videos/{slug}-{ID}xhamster.com/xembed.php?video={ID}&autoplay=1xvideos.com/video{ID}/xvideos.com/embedframe/{ID}
Critical: Use /embed/ paths, never /videos/ paths. Watch page URLs loaded in iframes will not honor autoplay.
Required iframe attributes:
allow="autoplay; encrypted-media"
Query parameters on all embeds:

autoplay=1 — tells embedded player to start immediately
mute=1 — required because browsers block unmuted autoplay

3.3 Fallback
If getEmbedUrl() has no matching transform rule for a domain, return the URL as-is (some sites work in iframes without transformation). If the site blocks iframing via X-Frame-Options, show an error state with option to open in new tab.

4. Auto-Advance System
4.1 Timer-Based Advance
Since cross-origin iframes block playback events, auto-advance is timer-based.
Algorithm:

Each video has a duration field (string like "7:00" or "12:34")
Parse to seconds: parseDuration("7:00") → 420
When a video starts playing, set countdown = durationSeconds + bufferSeconds
Default buffer: 5 seconds
Countdown decrements every 1 second
When countdown reaches 0: advance currentIndex to next video
React re-renders, iframe gets new src with next embed URL
If end of queue: wrap to index 0 (loop) or stop (based on user setting)

4.2 Timer Limitations
The timer cannot detect:

If the user pauses inside the iframe
If the user skips ahead inside the iframe
If the video actually finished early or late
If the listed duration is incorrect

4.3 User Controls for Timer

Manual skip: Previous / Next buttons bypass the timer
Adjust buffer: Per-source configurable buffer (FapHouse might need +3s, BoyfriendTV might need +8s)
"Video ended early?" button: Immediately skips to next
"+30s / -30s" buttons: Adjust timer mid-playback
Duration correction: Save corrected durations so timer is accurate on replay
Pause countdown: Pause the countdown without pausing the video (for rewatching)
Countdown display: Always visible, shows time until next video
Auto-advance toggle: Turn timer off entirely for manual-only control

9. Playback Features Spec
9.1 Scene Intelligence
Purpose: Replace blind scrubbing with clickable scene blocks.
Data model per video:
scenes: [
  { start: 0, end: 150, label: "Intro", thumbnail_url: "..." },
  { start: 150, end: 315, label: "Scene 1", thumbnail_url: "..." },
  { start: 315, end: 540, label: "Scene 2", thumbnail_url: "..." },
]
Display: Horizontal blocks below the player, proportionally sized by duration. Tapping a scene sets the video embed URL's t parameter (e.g., ?autoplay=1&mute=1&t=150) and reloads the iframe.
Note: The t parameter works on some embeds (FapHouse) but not all. Where unsupported, scene markers are informational only.
9.2 Mood Lanes
Structure: Up to 3 parallel playlists (lanes) loaded simultaneously.
SessionState {
  lanes: [
    { id: "lane-1", name: "Main", queue: Video[], currentIndex: number },
    { id: "lane-2", name: "Alt", queue: Video[], currentIndex: number },
    { id: "lane-3", name: "Wild", queue: Video[], currentIndex: number },
  ],
  activeLane: "lane-1"
}
Switching lanes changes which queue drives the player. Position in inactive lanes is preserved.
9.3 Quick Fade
Trigger: Double-tap, three-finger tap, or remote button press.
Behavior:

Iframe is hidden (not destroyed — preserves playback state)
Clean dashboard overlay appears (NOOG logo, session stats, non-explicit)
Timer pauses
Same gesture again dismisses overlay, shows iframe, resumes timer

Session stats on cool-down screen:

Videos watched count
Clips saved count
Session duration
Current queue position

9.4 Clip Saving
Flow:

User activates clip mode
Sets start time and end time (manual input or "mark start" / "mark end" buttons synced to countdown)
Optional label
Clip is saved to database
On replay of the same video, clips are shown as highlighted regions on the scene timeline
"Play All Clips" creates a temporary playlist that loads each clip's video at the clip's start timestamp

Limitation: Since we can't control the embedded player's position precisely, clip playback sets the t parameter on the embed URL. The start is approximate.
9.5 Pre-Queue (Auto-Generated Next Queue)
Trigger: When current queue reaches the last video, NOOG generates a suggested next queue.
Algorithm:

Analyze tags, categories, and sources of videos watched in current session
Query AI assistant or recommendation engine for 8–10 similar videos
Present as "Your queue ended. We built one based on tonight's session."
User can approve entire queue, remove individuals, or dismiss

13. Room System Architecture
13.1 How Rooms Extend the Remote
The remote control system pairs one TV with one phone via Socket.IO. Rooms open the same socket room to multiple connections. State management is identical (queue, current index, play/pause state, countdown). The only addition: multiple clients can push events instead of just one controller.
13.2 Room Lifecycle

Host creates room → gets code (same format: NOOG-XXXX)
Host shares code or QR
Participants join via code
Host's player is the "source of truth" for what's playing
Timer runs on host's device (or server-side for reliability)
All participants see the same current video via synced state
Queue additions from participants are broadcast to all
Host closes room → room:closed event → all participants disconnected
